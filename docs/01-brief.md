# Brief: Исследование рынка и архитектурное решение для системы архивации электронной почты

**Документ:** 01-brief  
**Дата:** 2026-07-06  
**Статус:** Утверждено

---

## 1. Постановка проблемы

Пользователю требуется единое self-hosted приложение для архивации электронной почты, которое:
1. Позволяет добавлять произвольные почтовые аккаунты (IMAP).
2. Архивирует письма в фоновом режиме с гибкой настройкой правил.
3. Обеспечивает полнотекстовый поиск по архиву.
4. Позволяет восстанавливать письма обратно в почтовые ящики.
5. Реплицирует архив в S3-совместимое облачное хранилище.

Существующие решения не удовлетворяют этим требованиям в комплексе.

---

## 2. Исследование рынка

### 2.1. Проанализированные решения

| Решение | Тип | Проблемы |
|---------|-----|----------|
| **Mail-Archiver** | Self-hosted, .NET + PostgreSQL | Требует Docker и .NET-экосистему. Нет восстановления через IMAP APPEND. Завязан на хостера. |
| **Open Archiver** | Self-hosted, Node.js + SvelteKit + Meilisearch + PostgreSQL | 4+ контейнера (Redis, Meilisearch, PostgreSQL, Node). Перегружено для персонального использования. Нет единого бинарника. |
| **Eonvelope** | Python | Python GIL — узкое место при параллельной синхронизации множества ящиков. Нет единого сервиса. |
| **MailVault** (open source) | macOS/Linux | Хранит в .eml, но нет полнотекстового индекса, нет правил архивации, нет S3. |
| **MailStore Home** | Проприетарный, Windows only | Проприетарная БД, только Windows. На Mac/Linux — никак. Нет S3. |

### 2.2. Общие боли рынка

1. **SaaS-зависимость** — большинство решений требуют облачного API (Google Workspace, Microsoft 365) и не умеют работать с произвольным IMAP.
2. **Docker-обязаловка** — даже для локального запуска нужен Docker Compose с 3-5 сервисами.
3. **Нет гибких правил** — либо архивируется всё подряд, либо ничего. Нельзя задать: «архивируй только письма от @client.com с вложениями».
4. **Восстановление — отдельная боль** — многие экспортируют в .mbox, но не умеют вернуть письма обратно в ящик через IMAP.
5. **Нет S3-репликации** — ни одно из найденных open-source решений не предоставляет встроенной асинхронной зеркалирования в S3 с client-side шифрованием.
6. **Проприетарные форматы** — MailStore, прочие enterprise-решения хранят данные в закрытых БД, .eml извлечь невозможно.

## 3. Архитектурное решение

### 3.1. Выбор технологий

| Компонент | Технология | Обоснование |
|-----------|-----------|-------------|
| **Язык** | Go 1.25+ | Goroutines для параллельной синхронизации множества ящиков; один бинарник без зависимостей. |
| **IMAP** | `emersion/go-imap` | Лучшая библиотека для Go. Поддерживает IDLE, UIDPLUS, CONDSTORE. |
| **SMTP** | `go-mail/mail` | Fallback для восстановления, если IMAP APPEND не поддерживается. |
| **БД метаданных** | SQLite (`modernc.org/sqlite`, чистый Go) | Zero-config, один файл, WAL mode. Не требует отдельного процесса. |
| **Поиск** | `blugelabs/bluge` | Встраиваемый движок на чистом Go. Не требует Meilisearch/Elasticsearch. |
| **S3** | `aws-sdk-go-v2` | Стандарт, поддерживает AWS, MinIO, Yandex, Wasabi, Ceph. |
| **Шифрование** | Argon2id + AES-256-GCM | Master Key для credentials и client-side encryption S3. |
| **HTML Sanitizer** | `microcosm-cc/bluemonday` | Очистка HTML писем перед рендерингом. |
| **Web UI** | HTMX + Tailwind + Go templates | Минимум JS, максимум скорости разработки, один бинарник. |
| **Scheduler** | `robfig/cron` + `panjf2000/ants` | Cron для расписания, пул goroutines для ограничения concurrency. |

### 3.3. Ключевые потоки данных

**Поток 1: Архивация**
1. Scheduler запускает Sync Engine через goroutine pool.
2. IMAP Client подключается, получает список папок.
3. Для каждой папки: сравниваем `UIDVALIDITY` с SQLite. Если изменился — полная ресинхронизация.
4. Запрашиваем UID-ы выше последнего заархивированного.
5. Rule Engine проверяет письмо. Если `skip` — пропускаем.
6. Письмо записывается в локальный Maildir (`tmp/` → `new/`).
7. Bluge индексирует метаданные и содержимое (имена вложений, не содержимое).
8. Метаданные записываются в SQLite через Single Writer Pattern.
9. S3 Async Uploader ставит файл в очередь `s3_upload_queue` (SQLite).
10. Только после успешных шагов 6-8 UID фиксируется. При ошибке — транзакция откатывается.

**Поток 2: Просмотр письма**
1. Web UI запрашивает письмо по ID.
2. Backend проверяет SQLite: `storage_location = local | s3`.
3. Если `local` — читаем .eml из Maildir.
4. Если `s3` — проверяем LRU-кэш. Если нет — скачиваем из S3, расшифровываем, пишем в кэш.

**Поток 3: Восстановление**
1. Пользователь выбирает письмо(а) и целевой ящик.
2. Backend проверяет локальное наличие. Если нет — приоритетный lazy load из S3 (не вытесняет кэш).
3. IMAP Client подключается к целевому ящику.
4. Выполняется `IMAP APPEND` с оригинальным `INTERNALDATE` и флагами.
5. Если APPEND не поддерживается — fallback на SMTP.
6. Результат записывается в `restore_logs`.

**Поток 4: Retention (фоновый)**
1. Cron-задача `retention` запускается ежедневно в 03:00.
2. Для писем с истёкшим `retention_local_days` и `storage_location = local`:
   - Загружаем в S3 (если ещё не загружено).
   - Удаляем локальный .eml.
   - Обновляем `storage_location = s3`.
3. Для писем с истёкшим `retention_s3_days`: удаляем из S3 и SQLite каскадно.

## 4. Логические нестыковки, выявленные при проектировании, и их решения

### 4.1. SQLite + множественные sync-горутины

**Проблема:** SQLite в WAL mode поддерживает множественных readers, но writer один. При 10 параллельных IMAP-синхронизациях возникает contention на запись метаданных.

**Решение:** Внедрён **Single Writer Pattern**. Все INSERT/UPDATE/DELETE в SQLite проходят через единый канал Go и одну goroutine-писатель. Остальные компоненты (Sync Engine, S3 Uploader, Rule Engine) отправляют запросы через канал. Это гарантирует отсутствие deadlock и WAL contention.

### 4.2. Дедупликация писем

**Проблема:** Одно письмо может быть в нескольких папках/аккаунтах. Хранение через hard links/symlinks ломается на Windows и усложняет backup.

**Решение:** Локально — каждый аккаунт/папка имеет свой файл .eml (без дедупликации). Дедупликация выполняется только на уровне S3: object key строится по SHA-256 содержимого. Одинаковые письма заливаются один раз. SQLite хранит множественные ссылки на один S3-объект.

### 4.3. Bluge-индекс vs S3-репликация

**Проблема:** Если индекс строится после загрузки в S3, а письмо затем вытеснено локально — индекс может указывать на несуществующий локальный файл.

**Решение:** Bluge индексирует письмо **на этапе получения** (шаг 7 потока 1), до асинхронной загрузки в S3. Индекс содержит `email_id`, а не путь к файлу. Просмотр S3-писем — через lazy load по `email_id`, без переиндексации.

### 4.4. Восстановление vs LRU-кэш

**Проблема:** Письмо вытеснено в S3, кэш переполнен. Восстановление может быть заблокировано, если LRU вытеснит только что скачанный файл.

**Решение:** Lazy load при восстановлении — **приоритетный запрос**. Скачанный файл помечается как «не вытесняемый» на время операции восстановления. После успешного IMAP APPEND — файл возвращается в обычный LRU-режим.

### 4.5. Master Key и S3 Encryption

**Проблема:** Разные ключи для credentials и S3 усложняют управление. Потеря одного из них делает данные недоступными.

**Решение:** **Единый Master Key** (Argon2id от пароля пользователя). Он используется для:
- Шифрования IMAP-паролей и OAuth2 токенов в SQLite.
- Client-side encryption файлов перед загрузкой в S3 (через HKDF-производный ключ).

Master Key не хранится на диске и не передаётся в S3. Потеря Master Key = полная потеря доступа к зашифрованным данным в S3. Система выводит явное предупреждение при первой настройке.

### 4.6. Retention vs Mirror

**Проблема:** Если retention вытесняет письмо в S3 до того, как оно туда загружено — данные теряются.

**Решение:** Чёткая последовательность:
1. **Mirror** — письмо сразу архивируется локально и ставится в очередь на S3.
2. **Retention** — только после подтверждённой загрузки в S3 (`s3_etag` не NULL) локальная копия может быть удалена.
3. Cron-задача retention проверяет `s3_etag` перед удалением локального файла.

### 4.7. HTML-рендеринг vs XSS

**Проблема:** Просмотр HTML-писем в браузере — классическая XSS-атака. «DOMPurify-подобный sanitizer» — неопределённость.

**Решение:** Используется конкретная библиотека `microcosm-cc/bluemonday` (Go-нативная). Удаляются script, onclick, внешние ресурсы, data URI. Дополнительно CSP-заголовок: `default-src 'self'; script-src 'self';`.

### 4.8. IMAP APPEND vs Message-ID

**Проблема:** Некоторые серверы отклоняют APPEND с дублирующимся Message-ID. «Восстановить как новое» — дополнительная логика.

**Решение:** В MVP восстановление выполняется **только с оригинальным Message-ID**. Если сервер отклоняет — операция помечается как `failed`, ошибка логируется в `restore_logs`. В бэклог v2 вынесена фича «восстановить с новым Message-ID».

### 4.9. S3 Queue vs SQLite транзакции

**Проблема:** Очередь S3-загрузок в SQLite (`s3_upload_queue`) может конфликтовать с записью метаданных писем.

**Решение:** Таблица `s3_upload_queue` включена в общую схему SQLite и обслуживается тем же Single Writer Pattern. Worker S3-загрузки читает из очереди (SELECT), загружает в S3, затем отправляет UPDATE через канал писателя. Нет прямого доступа к БД из воркеров.

### 4.10. Config YAML vs Web UI Rules

**Проблема:** Правила можно менять через Web UI и через YAML. Возможен race condition.

**Решение:** Web UI пишет напрямую в SQLite (через Single Writer). YAML (`rules.yaml`) мониторится через `fsnotify` или таймер раз в 30 секунд. При изменении YAML — перезагрузка в SQLite. Источник истины — SQLite. YAML — удобство для power users, но не обход Web UI.

## 5. Выводы

Предложенная архитектура решает все ключевые боли пользователя:
- **Единый сервис** — один бинарник, zero-dependency.
- **Произвольные IMAP-аккаунты** — не завязаны на хостера.
- **Гибкие правила** — AND/OR, regex, retention, действия.
- **Полнотекстовый поиск** — встроенный Bluge, без Meilisearch.
- **Восстановление** — IMAP APPEND + fallback SMTP.
- **S3-репликация** — асинхронная, с client-side шифрованием.
- **Открытый формат** — Maildir (.eml), читается любым клиентом через 20 лет.

Все выявленные логические нестыковки устранены конкретными архитектурными решениями (Single Writer, приоритетный lazy load, SHA-256 дедупликация в S3, единый Master Key).
